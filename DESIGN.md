## Problem Statement

每天手动打开深圳市住建局官网，逐条翻通知公告、政策法规、土地出让栏目，判断哪些值得做成短视频——这个过程每天吃掉 20-30 分钟。信息分散在多个栏目、多个页面，没有聚合、没有去重、没有自动存档。

需要构建一条自动采集流水线：定时抓取官网更新 → 截图存档 → 结构化提取内容 → 写入飞书多维表格 → 你勾选"可做"的选题 → 自动生成短视频口播文案 → 排入 video_factory 发布队列。

## What Makes This Cool

1. **醒来即看** — 每天早上多维表格里已经有昨日/今晨的新增信息，带截图，你 2 分钟扫完决定今天做什么
2. **策展权归你** — AI 不替你做内容判断，它只负责采、存、写初稿。勾选哪个、怎么讲、什么角度，你定
3. **与 video_factory 无缝衔接** — 勾选 → 生成文案 → 排期发布，一条流水线走到底
4. **不怕官网改版** — HTML 解析主力 + AI 视觉兜底的双轨制，政府网站改版是常态，这套架构吃得下

## Prior Art

本项目建立在两条已有体系之上：

- **property-clawer**: 深圳房产成交数据抓取系统，覆盖政府开放数据平台 API 调用、SQLite 存储、图表生成、飞书通知。已有成熟的 `BaseFetcher` 模式和 Cron 调度。
- **video_factory**: 视频自动化发布流程，覆盖飞书多维表格管理、TTS 视频生成、视频号发布的浏览器自动化。已有 Cron 脚本框架和 `lark-cli` 集成。

本次新增的"官网监控模块"填补了数据源侧的缺口——之前只有结构化 API 数据（成交套数、面积），缺少政策/土拍/规则这类"故事型"素材。

## Premises

1. **官网内容 = 短视频素材金矿** — 政策条文、土拍公告这些东西本身就自带话题性，适合"信息解读"类短视频
2. **官网更新频率低** — 一天几条新内容就不错了，不存在"信息过载"问题，所以全量抓取是合理的
3. **官网结构相对稳定** — 政府网站虽然会改版，但栏目分类（通知公告、政策法规、土地出让）的逻辑不会变
4. **HTML 解析为主、AI 视觉兜底是正确策略** — 结构化解析快且准，AI 视觉慢但鲁棒，两者互补
5. **策展权必须在人手里** — AI 能判断"这条是不是讲土拍"，但判断不了"这条值不值得做视频"

## Approaches Considered

### Approach A: 轻量 Cron 脚本

**Summary**: Python + Playwright 定时访问栏目页 → CSS 选择器提取标题/日期/链接 → 截图 → lark-cli 写入多维表格。一条脚本全搞定。

**Effort**: S (1-2天)
**Risk**: Medium（官网改版时 CSS 选择器失效）
**Pros**:
- 极简，一个 .py 文件 + 一个 cron 条目即可运行
- 开发速度最快，今天写明天跑
- 无外部依赖，不依赖 browse daemon 或 AI 服务
**Cons**:
- CSS 选择器是单点故障——政府网站平均每年至少改版一次
- 截图只是整页截图，没有元素级截图（如只截"正文区域")
- 内容提取质量完全依赖 HTML 结构质量，遇到表格/PDF 附件无能为力
**Reuses**: 现有 Cron 调度框架

### Approach B: gstack browse 全量驱动

**Summary**: 用 gstack browse 无头浏览器进行定时访问 → 截图（支持全页+元素级）→ AI 视觉模型直接"看"截图提取结构化信息 → lark-cli 入库。

**Effort**: M (3-4天)
**Risk**: Low-Medium（依赖 browse daemon 常驻稳定性）
**Pros**:
- 天然抗改版：AI 视觉识别不依赖 HTML 结构，官网怎么改都能提取
- 截图质量最佳：browse 工具支持元素级截图、响应式截图
- 可处理图片型公告（有些政府公告是扫描件图片）
**Cons**:
- 依赖 browse daemon 7x24 运行，daemon 挂了采集就停
- AI 视觉识别成本高于 HTML 解析
- 需要额外处理 browse session 管理和 cookie 维护
**Reuses**: gstack browse 基础设施

### Approach C: 混合架构（推荐）

**Summary**: HTML 解析做主力提取（快、省），截图做视觉证据（全页+正文区域双层截图），AI 视觉做兜底（解析失败或检测到页面结构变化时自动切换）。新增去重逻辑（按 URL/标题哈希判断是否已入库）。与现有 video_factory Cron 体系统一调度。

**Effort**: M (2-3天)
**Risk**: Low
**Pros**:
- 两种提取路径互为备份，单点不失效
- HTML 解析的"结构变化检测"可自动触发 AI 兜底，无需人工发现
- 去重逻辑避免重复入库浪费你的策展时间
- 与现有 Cron 统一管理，不引入新的调度系统
- 全页截图（用于视频素材画面）+ 正文区域截图（用于快速浏览）双截图
**Cons**:
- 比 A 方案多 1-2 天开发量
- 需要同时维护 CSS 选择器和 AI prompt 两套提取逻辑
- 截图存储空间会增长（每天 ~5-20MB），需要定期清理策略
**Reuses**:
- property-clawer 的 BaseFetcher 模式
- video_factory 的 Cron 框架 + lark-cli 集成
- lark-base skill 的多维表格字段管理
- lark-drive skill 的截图文件上传

## Recommended Approach

**选择 Approach C: 混合架构（评审加固版）**

理由：
1. 你已有成熟的 Python Cron 体系（property-clawer + video_factory），混合架构是自然延伸
2. 政府网站改版不是"会不会"的问题，是"什么时候"的问题——双轨制是必要投入
3. 去重逻辑虽然多写几十行，但从 day 1 就避免了"同一份公告入库三次"的体验伤害
4. 开发量的增量换来的是"设完就忘"的可靠性

**评审驱动的关键加固**（详见末尾 GSTACK REVIEW REPORT）：
- 浏览器生命周期管理（context manager + 僵尸进程清理）
- 反爬对策（stealth + 随机延迟 + CAPTCHA 检测告警）
- 本地暂存 + 重试（SQLite 先存再推飞书，3 次指数退避重试）
- Cron 环境硬化的 wrapper 脚本（完整 PATH + 环境变量 + 日志）
- 每次执行发心跳通知（静默 ≠ 成功）
- 每天 2 次执行（8:00 + 18:00）
- PDF 作为 Phase 1 一等需求（pdfplumber 提取文本）
- 文案生成改为"事实+影响+问题"辅助模式（非全自动写稿）
- 完整测试套件 + HTML fixtures

## Implementation Plan（评审修订版，5-8 天）

### Phase 0: 前置验证（0.5 天）— 编码前必做

1. 手动检查目标网站 RSS/监控服务可用性（2h）
2. 保存一份真实页面 HTML 到 `tests/fixtures/`（0.5h）
3. 确认 lark-cli Drive upload → Base attachment 格式兼容（0.5h）

### Phase 1: 爬虫核心 + 反爬 + PDF（2-3 天）

1. 确认住建局+规自局各栏目 URL 结构
2. 编写 `gov_crawler.py`：
   - Playwright context manager + 僵尸进程清理 + 预检
   - stealth 补丁 + 随机延迟 5-15s
   - CAPTCHA 检测（"验证"/"滑块"关键词）→ 飞书告警
   - 栏目页 → 条目列表提取（标题/日期/链接）
3. PDF 附件处理：检测 .pdf → 下载 → pdfplumber 提取文本 → 标记
4. 去重：URL 标准化 + 标题归一化 → gov_cache.db
5. 截图函数（全页 + 正文区域，统一路径）
6. 结构变化检测：3 信号 2-of-3 触发 AI 视觉（30 天滚动基线）
7. AI 视觉兜底：截图 → AI 视觉提取 → cross-validate（link count 比对）

### Phase 2: 飞书入库 + 本地暂存 + 重试（1-1.5 天）

1. 新建/复用多维表格 Base，创建字段（含新增字段）
2. 本地暂存：SQLite 先存，标记 pending/synced
3. `feishu_writer.py`：lark-cli 写入 + 3 次指数退避重试
4. 截图上传（Drive upload → attachment，格式验证）
5. 心跳通知："扫描 N 栏目，新增 M 条，错误 E 条"

### Phase 3: 创作素材生成（1.5-2 天）

1. `gov_script_generator.py`：扫描"入选" → 调 AI → 生成素材（非全稿）
2. AI 视觉兜底完善：交叉验证 + ai_fallback 标记 + 原始响应日志
3. 输入安全：截断 4000 字符 + 系统提示词模式过滤
4. 与 video_factory Cron 体系统一调度

### Phase 4: 测试 + Cron 部署 + 监控（1-2 天）

1. Cron wrapper 硬化：完整 PATH + 环境变量 + 绝对路径 + 日志
2. 部署 Cron：每天 8:00 + 18:00（SHELL=/bin/zsh）
3. Watchdog：8:15 心跳检查 → 缺失告警
4. 测试套件：fixture 回放 + 去重幂等 + 飞书 mock + e2e
5. 实际跑 3-5 天，调参去重阈值和结构变化检测基线

#### CEO Finding 1: Problem Framed Around Time Savings, Not Content Performance (CRITICAL)

The plan's problem statement says "this process eats 20-30 minutes daily." Saving 30 min/day saves ~3h/week worth ~$150. Against a 2-3 day build, breakeven takes months. More importantly, the plan never connects the pipeline to audience metrics: subscriber growth, view counts, engagement, revenue.

The plan silently bets that government policy announcements perform as well as current content. Zero evidence. Many creators who built "news automation" discovered the content underperforms because audiences want personality, not aggregation.

#### CEO Finding 2: AI Summary Gates Curation (HIGH)

The human curator sees AI-generated summaries, not original content. If the summary misses the hook, content never gets selected. If the summary hallucinates, wrong content gets produced. The "human in the loop" is only as good as the AI filter they read through.

### ENG REVIEW — Full Findings

#### Eng Finding 1: Playwright Browser Lifecycle Unmanaged (CRITICAL)

The architecture says "Playwright opens page." Playwright launches a persistent Chromium process. Zero mention of: launch/close strategy, crash recovery, zombie process cleanup, binary update incompatibility. The existing codebase uses `requests` + `BeautifulSoup` — the new crawler introduces a different browser automation stack with no pattern to follow.

**Fix:** Use `with sync_playwright() as p:` context manager. Add `try/finally` that kills orphaned chromium by PID. Add pre-flight `playwright install chromium` check. Add 30s launch timeout with 3 retries.

#### Eng Finding 2: Subprocess Shell Injection via lark-cli (HIGH)

`daily_news_crawler.py` passes crawled content into `subprocess.run` via `--json`. AI-generated text is uncontrolled — if lark-cli shells out internally, backticks/`$()` metacharacters become arbitrary command execution.

**Fix:** Use `--json-file` (if supported) or Feishu HTTP API directly via `requests`. At minimum, validate no shell metacharacters before subprocess.

#### Eng Finding 3: No Retry or Partial-Failure Semantics (HIGH)

If Feishu API is down for 5 minutes at 8:00, the entire day's crawl data is discarded. No retry. No local cache. Specific gaps: Feishu 429/503, partial column failures, Playwright timeout on one article, screenshot success but text failure, lark-cli not in PATH under cron.

**Fix:** Add local-staging in SQLite. Pipeline becomes: fetch → store locally → attempt Feishu write with 3 retries + exponential backoff → mark synced. `--retry-failed` mode replays unsynced records.

#### Eng Finding 4: Government Site Anti-Bot Measures Unaddressed (HIGH)

.gov.cn sites deploy: TLS fingerprinting (JA3), behavioral analysis, CAPTCHA walls (especially during policy-sensitive periods), IP rate limiting, Alibaba Cloud WAF. A CAPTCHA would load successfully (HTTP 200), pass structure change detection (has DOM), but AI vision would "see" a CAPTCHA. The system silently reports "today: 0 new."

**Fix:** Add `playwright-stealth`, randomized delays (5-15s), CAPTCHA detection (check for "验证"/"滑块"), Feishu alert on CAPTCHA detection, real Chrome profile option, `--headed` debug mode.

#### Eng Finding 5: Structure Change Detection Undefined (HIGH)

Phase 1 step 5 is one sentence describing what will consume disproportionate maintenance time. Single-threshold "hit rate" conflates structure change with slow news day. Pagination breaks it. No historical baseline.

**Fix:** Multi-signal check — (1) CSS hit count outside 30-day rolling range, (2) page text fingerprint hash, (3) DOM depth/element count stats. 2-of-3 triggers AI fallback + alert. On success, auto-update selectors (self-healing).

#### Eng Finding 6: Screenshot Path Inconsistency (MEDIUM)

Three different paths: `~/Videos/screenshots/`, `~/video_factory/screenshots/`, `~/property_news/screenshots/`. Pick one.

**Fix:** Use `~/Projects/property_news/screenshots/` consistently.

#### Eng Finding 7: Cron Environment Will Silently Break Everything (HIGH)

macOS cron runs with `PATH=/usr/bin:/bin`, no env vars, no keychain access, no Full Disk Access. The existing crontab is nearly empty. Every new Python cron job hits this.

**Fix:** Cron wrapper must set full PATH, source env vars, activate venv, use absolute paths, redirect to dated log file, set `SHELL=/bin/zsh`.

#### Eng Finding 8: Dedup Logic Vulnerable to Title Variations (MEDIUM)

Chinese government titles break naive hashing: full-width vs half-width spaces, Chinese vs English punctuation, cross-column duplicates, URL variations.

**Fix:** Normalize before hashing. URL as primary key, normalized title as fallback. Store both raw and normalized forms.

#### Eng Finding 9: No Test Plan (HIGH)

Phase 4 is "run for 3-5 days manually" — monitoring, not testing. Missing: dedup idempotency, empty page handling, Feishu API failure injection, malformed HTML, screenshot failure, encoding edge cases, date parsing variants.

**Fix:** Create `tests/` with fixtures (saved HTML snapshots), test_dedup.py, test_extraction.py, test_screenshot.py, test_feishu_write.py, test_end_to_end.py.

#### Eng Finding 10: AI Vision Fallback Has Undetected Failure Modes (MEDIUM)

Vision models hallucinate titles, fabricate dates, miss edge items, cost money per call, and produce unverifiable output.

**Fix:** Cross-validate vision vs simple `len(soup.find_all('a'))`. Flag `ai_fallback=True` in Base. Log raw AI response. Budget ~$0.10-0.50/month for fallback calls.

#### Eng Finding 11: Feishu Attachment Upload Complex (MEDIUM)

Base attachment field format may differ from Drive upload return format. Phase 2 step 3 ("lark-cli drive upload → attachment field") may silently fail.

**Fix:** Before Phase 2, manually test: upload screenshot → get file token → include in record-upsert → verify clickable in Feishu UI.

#### Eng Finding 12: PDF Is First-Class, Not Optional (MEDIUM)

Government bureaus publish formal notices, policy documents, and land auction announcements as PDFs. If 30% of content is PDF-only, the system silently misses 30%.

**Fix:** Commit to PDF strategy in Phase 1. Download PDFs, extract via `pdfplumber`/`PyMuPDF`, flag scanned docs for OCR. At minimum, detect and flag PDF-only items.

#### Eng Finding 13: No Silence-Means-Failure Monitoring (MEDIUM)

If the script crashes at 8:01, you get no notification. If the site returns a maintenance page, you get "today: 0 new" — indistinguishable from a genuinely quiet day.

**Fix:** Send heartbeat on every run. Watchdog cron at 8:15 checks heartbeat exists. Add `/health` check for Playwright + lark-cli + site reachability. "0 new" messages include confidence indicator.

#### Eng Finding 14: Single Daily Run Is Fragile (MEDIUM)

Government sites update unpredictably. If site is down for maintenance at 8am (common for .gov.cn), entire day is lost.

**Fix:** Run twice daily: 8:00 + 18:00. Dedup already handles duplicates. Doubles reliability at near-zero cost.

#### Eng Finding 15: Content Safety Before AI Processing (LOW)

Crawled content sent directly to AI for script generation. If government site is ever compromised, malicious prompt injection could influence AI output.

**Fix:** Truncate to 4000 chars. Strip lines matching AI system prompt patterns.
